Quero implementar a PARTE 2 do módulo “Base de Clientes”:
a tela de consulta manual, onde qualquer usuário (vendedor, coordenador, master) pode digitar CPF ou matrícula e ver os dados da nossa base.

Tecnologia do projeto:

Backend: Node.js + Express (pasta server/)

Banco: PostgreSQL (já com as tabelas da PARTE 1)

Frontend: React (pasta client/)

As tabelas já existentes (da Parte 1) e que devem ser usadas são:

clientes_pessoa

clientes_folha_mes

clientes_contratos

Não altere a estrutura delas, apenas use.

🎯 OBJETIVO FUNCIONAL

O usuário informa CPF ou matrícula.

O sistema localiza a pessoa na base.

A tela mostra, de forma organizada:

Dados principais do cliente (uma única vez)

Situação de folha (margens, líquido, etc.)

Todos os contratos encontrados (agrupados por pessoa)

Se o mesmo CPF tiver mais de uma matrícula, a tela deve permitir escolher qual matrícula visualizar.

Não é para consultar API externa ainda (Lemit).
Parte 3 será só higienização. Aqui é só base interna.

1. BACKEND – ENDPOINTS DE CONSULTA

Quero dois endpoints:

1.1. GET /api/clientes/consulta

Função: busca inicial por CPF ou matrícula, trazendo uma visão resumida para decisão.

Query params:

cpf (opcional)

matricula (opcional)

Regras:

Se vier matricula, a busca é direta pela coluna clientes_pessoa.matricula.

Se vier cpf, buscar todas as pessoas com esse CPF.

Se vierem os dois, priorizar matrícula.

Se não vier nenhum, retornar 400.

Retorno esperado (JSON):

{
  "tipo_busca": "cpf" | "matricula",
  "termo": "valor que o usuário digitou",
  "resultados": [
    {
      "pessoa_id": 123,
      "cpf": "00000000000",
      "matricula": "1234567",
      "nome": "NOME DO CLIENTE",
      "convenio": "SIAPE",
      "orgao": "ORGÃO XYZ",
      "uf": "SP",
      "municipio": "SÃO PAULO",
      "sit_func": "ATIVO"
    }
  ]
}


Comportamento:

Se não encontrar nada → resultados: []

Se encontrar várias pessoas para o mesmo CPF → retornar todas no array (para o frontend deixar o usuário escolher qual matrícula abrir).

Se encontrar uma só → mesmo assim retornar num array (frontend trata).

1.2. GET /api/clientes/:pessoaId

Função: busca detalhada de um cliente específico (após escolher matrícula ou vindo direto pela matrícula).

Regras:

Recebe pessoaId (id da tabela clientes_pessoa).

Busca os dados em:

clientes_pessoa (dados fixos)

clientes_folha_mes (todas as competências, ordenadas da mais recente para a mais antiga)

clientes_contratos (todos os contratos da pessoa)

2. FRONTEND – TELA DE CONSULTA MANUAL

Quero uma tela em React na área do CRM chamada “Consulta de Cliente” com o seguinte comportamento:

2.1. Parte superior – Barra de busca

Campo para o usuário digitar:

CPF (com máscara) ou

Matrícula

Pode ser:

um campo com toggle “Buscar por CPF / Buscar por Matrícula”

ou dois campos, mas SOMENTE um deles é enviado a cada busca.

Botão: “Consultar”.

Estados visuais:

Enquanto chama /api/clientes/consulta, mostrar um loading (“Buscando cliente na base…”).

Se não encontrar nada: mostrar mensagem clara:

“Nenhum cliente encontrado para esse CPF/matrícula.”

2.2. Caso 1 – Apenas 1 resultado

Se resultados.length === 1 na resposta de /api/clientes/consulta:

Disparar automaticamente uma nova requisição para /api/clientes/:pessoaId

Abrir a visualização detalhada (descrita abaixo).

2.3. Caso 2 – Vários resultados (CPF com múltiplas matrículas)

Se resultados.length > 1:

Mostrar uma lista / tabela com cartões resumidos:

Nome

CPF

Matrícula

Convênio

Órgão

Situação funcional

Cada linha/card tem um botão “Ver detalhes”.

Ao clicar em “Ver detalhes”, chamar /api/clientes/:pessoaId e exibir a visualização detalhada.

2.4. Visualização Detalhada (após /api/clientes/:pessoaId)

Organizar a tela em três blocos:

Nome

Natureza

Bloco 3 – Contratos

3. REGRAS DE PERFORMANCE

As consultas SEMPRE são por CPF ou matrícula, então:

Use índices em clientes_pessoa.cpf e clientes_pessoa.matricula.

As consultas detalhadas (/api/clientes/:pessoaId) vão trazer:

1 registro de pessoa

poucos registros em clientes_folha_mes (por pessoa)

alguns registros em clientes_contratos (também por pessoa)

Ou seja, é seguro carregar tudo num único endpoint sem paginação nesta fase.
