Objetivo

Criar o CRM de Vendas usando a Base de Clientes já importada, sem duplicar dados, permitindo:

Criar campanhas por filtro (sem reimportar clientes)

Distribuir leads por usuário e controlar remanejamento

Registrar interações que alimentam o Pipeline e permitem gestão (corretor, gestor, master)

Atualizar dados dinâmicos (margem, situação, contratos) com higienização periódica sem “somar” e sem duplicar

Princípios do modelo de dados

Cliente é único por CPF

Campanha não guarda “cliente duplicado”, guarda referência ao cliente

Dados dinâmicos são atualizados por “snapshot” e/ou “estado atual”

Contratos são sincronizados por chave e por substituição de conjunto (atualiza, adiciona, encerra)

Fluxo 1: Criar campanha sem reimportar
Opção principal (padrão)

Admin entra em “Base de Clientes / Compra de Lista”

Aplica filtros (convenio, banco, idade, margem, uf, etc)

Botão adicional: “Criar Campanha com esse filtro”

Sistema cria:

campaigns com filtros_json

campaign_leads com todos os client_id resultantes do filtro

Opção alternativa (quando precisar)

Importar campanha por CSV apenas com CPFs (ou CPF + campos dinâmicos)

Se for só CPF: vincula ao client existente e cria campaign_leads

Se vier CPF + margens/contratos atualizados: dispara fluxo de atualização (snapshot + sincronização) antes de criar os leads

Fluxo 6: Atualização de dados (higienização)

Objetivo
Atualizar dados dinâmicos sem duplicar e sem “somar contratos”.

Chaves

client: CPF

contract: numero_contrato + banco + tipo_produto (chave única)

Regras de importação de higienização

Upsert cliente por CPF

atualiza campos dinâmicos (situação, margens, lotação se vier)

mantém estáveis (nome, cpf)

contatos só adiciona novos

Snapshot

gravar client_snapshots com reference_date e fonte

Sincronização de contratos por substituição de conjunto
Para cada CPF:

lista_atual = contratos ativos no sistema

lista_arquivo = contratos do arquivo

Processo:

se contrato existe pela chave: update campos dinâmicos

se não existe: insert

se existe no sistema e não veio no arquivo: encerrar (status ENCERRADO + ended_at)
Isso evita virar 11 quando era 5 e depois veio 6

Remoção de contatos

nunca remover automaticamente pelo arquivo
Somente manual

Como a campanha deve lidar com dados atualizados

Cenário A: campanha criada por filtro interno

campanha referencia os clients atuais

se o cliente for atualizado depois, a campanha usa os dados mais recentes automaticamente

Cenário B: campanha criada após higienização externa
Duas opções

Atualizar base e depois criar campanha

recomendado, porque evita divergência

Criar campanha com snapshot (se quiser congelar margem daquele ataque)

salvar margem_snapshot e proposta_snapshot no campaign_leads

Recomendação padrão

atualizar base, depois criar campanha

snapshot no campaign_leads apenas se você quiser auditoria do que foi usado naquele ataque

O que não pode acontecer

Reimportar clientes completos a cada campanha

Duplicar contratos a cada atualização

Contrato que some do arquivo ficar “ativo” para sempre

Pipeline mostrar “Novo” e inflar volume sem interação

Métricas do pipeline baseadas só em quantidade, sem somar margens e propostas

Checklist de entrega para validar

Criar campanha via filtro em “Compra de Lista”

Distribuir em massa por quantidade por usuário

Admin ver “distribuídos” e remanejar

Atendimento replicado da Consulta Cliente, com Registrar Interação

Registrar interação exige marcador, contato usado, margem, proposta e observação

Meu Pipeline soma margens e propostas por coluna e não mostra “Novo”

Gestão Pipeline soma margens e propostas por marcador e abre pipeline do corretor ao clicar

Higienização atualiza snapshot e sincroniza contratos por substituição de conjunto
