1. Sistema de permissões flexível por usuário
1.1. Modelo de dados

Criar uma tabela (ou campos extras em users) para permissões granulares:

users

id

nome

email

tipo_usuario (enum simples pra referência: admin, master, coordenador, vendedor, atendimento, etc.)

user_permissions

id

user_id

module (string)

can_view (boolean)

can_edit (boolean opcional)

Sugestão de module (já pensando no futuro):

modulo_simulador

modulo_roteiros

modulo_base_clientes

modulo_compra_lista

modulo_crm_vendas_campanhas

modulo_crm_vendas_atendimento

modulo_academia

modulo_config_usuarios

modulo_config_precos

etc.

1.2. Tela de gestão de usuários

Na tela Administração > Usuários:

Ao editar um usuário, adicionar uma seção: Permissões de Acesso.

Mostrar uma lista de módulos com checkbox “Pode visualizar” (e se quiser, “Pode editar”).

O tipo de usuário (admin, master, etc.) pode servir só como “preset inicial”, mas a regra real passa a ser o que está em user_permissions.

1.3. Lógica de acesso

Backend: criar um middleware/helper que receba userId + module e valide se pode acessar.

Frontend: esconder itens do menu e telas inteiras se o usuário não tiver can_view naquele módulo.

Para o que você pediu agora:

CRM Vendas:

modulo_crm_vendas_campanhas → só admin, master, atendimento (por enquanto).

modulo_crm_vendas_atendimento → vendedores, coordenadores e quem mais você marcar.

2. CRM Vendas – Campanhas (importação + distribuição + remanejamento)
2.1. Distribuição por usuário (quantidade por usuário)

Na tela CRM Vendas > Campanhas, ao clicar em uma campanha:

Botão: Distribuir Leads

Abrir um modal com:

Select ou multiselect de usuários (apenas usuários que têm permissão de atendimento no CRM).

Campo “Quantidade por usuário” (ex: 50).

Mostrar o total disponível da campanha e quantos serão distribuídos.

Ao confirmar:

Para cada usuário selecionado, alocar X leads ainda “disponíveis”.

Atualizar contadores: “Disponíveis” e “Distribuídos”.

Modelo de dados (sugestão):

campaigns

id, nome, convenio, uf, status, total_leads, leads_disponiveis, leads_distribuidos

campaign_leads

id

campaign_id

cliente_id (ou referência à linha da base de clientes)

assigned_user_id (nullable)

status (disponivel, atribuido, em_atendimento, concluido, etc.)

data_atribuicao

data_ultimo_contato etc.

2.2. Visão de “Distribuídos” e remanejamento

Na listagem de campanhas (onde aparece “Total / Disponíveis / Distribuídos”):

Transformar o número de Distribuídos em um link/botão.

Ao clicar, abrir uma tela ou modal de Distribuição por usuário, mostrando:

Tabela:

Usuário

Quantidade de leads atribuídos

Quantos estão em:

Não trabalhados

Em atendimento

Concluídos (opcional)

Ações possíveis por linha:

“Remover X leads deste usuário e devolver à base da campanha”

“Transferir X leads deste usuário para outro usuário”

Regra de negócio:

Remanejar = atualizar assigned_user_id dos leads selecionados.

Devolver pra base = assigned_user_id = null e status = 'disponivel'.

4. Criar campanhas a partir da base de clientes (reaproveitar “Compra de Lista”)

Hoje você tem:

Base de Clientes > Compra de Lista

Filtros por convênio, órgão, UF, idade, situação funcional, etc.

Geração de “lista vendida” para terceiros.

Ideia:

No mesmo lugar, adicionar um botão: “Criar campanha a partir deste filtro”.

Fluxo:

O admin entra em Compra de Lista, monta os filtros normalmente.

Em vez (ou além) de “Simular / Comprar lista”, clica em Criar Campanha.

Abre modal:

Nome da campanha

Convênio (preenchido pelo filtro)

Observação (opcional)

O backend:

Salva um registro em campaigns com os parâmetros do filtro.

Cria registros em campaign_leads apontando para os clientes da base que batem com o filtro (sem duplicar dados).

Preenche total_leads, leads_disponiveis.

Benefícios:

Você não precisa ficar importando arquivo novo o tempo todo.

Usa a mesma base única, com todas as margens, contratos, etc.

Compra de lista e campanhas passam a ser “dois usos diferentes” da mesma base.
